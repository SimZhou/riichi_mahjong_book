# 日文网页翻译

## 概述

将日文网页内容翻译成自然、清楚、忠实的目标语言，同时保留原始结构与信息密度。

目标语言规则：

1. 如果用户明确指定目标语言，就按该语言输出
2. 如果用户没有指定，默认输出中文
3. 不要擅自把一次任务中的目标语言从中文切成英文，或从英文切回中文
4. 同一批页面内，术语、标题风格和语气必须稳定一致

输出必须保留：

1. 全部文本信息
2. 全部例题顺序
3. 全部图示和图片引用
4. 全部表格及其结构
5. 全部标题和分段边界
6. 全部强调信号，例如粗体、颜色强调、警示语和关键结论

不要加入代理式说明文字，例如“我将……”“下面进行翻译……”“这里我会……”。

## 必须遵循的流程

1. 先确认本次任务的目标语言。
2. 阅读原始日文 HTML 或提取出的正文内容。
3. 翻译前先识别所有承载内容的元素：
   - 标题
   - 段落
   - 列表
   - 表格
   - 行内牌图或插图
   - `../images/...` 下的独立图示
   - 各类强调内容，例如 `<strong>`、彩色文字、警示块、绿色总结块
4. 按原始顺序翻译，不要擅自合并分段，除非原文本来就是合并结构。
5. 保留全部例题块。原文有 `例1`、`例2`，就必须全部保留。
6. 保留全部图示引用。原文出现的文章图示，目标语言页中也必须出现。
7. 结构化保留表格。可以翻译表头和单元格，但不要把表格压缩成散文，除非目标格式确实无法承载表格。
8. 保留强调。原文如果通过粗体、颜色或总结块表达重要性，目标 markdown/HTML 也必须保留这种重要性。
9. 翻译完成后，再次对照原文检查是否有遗漏。

## 保真规则

以下都是硬性要求：

1. 不得遗漏任何段落、例题、条件、类似脚注的提醒或理论总结。
2. 不得把具体例子替换成概括性总结。
3. 不得因为牌例相似就省略重复例题。
4. 不得删除表格。
5. 不得删除图片块。
6. 不得删除行内的牌图示例。
7. 原文刻意强调的句子，不得弱化语气。
8. 不得凭空添加原文没有的新理论、新例题或解释性旁白。

## 翻译风格

目标语言表达应满足：

1. 准确优先
2. 可读性其次
3. 只有原文本来简洁时，才跟着简洁

优先采用：

1. 自然的目标语言教学式表达
2. 原文有明确逻辑连接时，在目标语言里也明确表达
3. 项目内稳定一致的目标语言麻将术语

避免：

1. 机器翻译味过重的措辞
2. 明明有清晰目标语言对应却残留日文术语
3. 代理式旁白
4. 为了“润色”而改变原始含义或信息密度

### 术语一致性

1. 目标语言一旦确定，章节名、牌理术语、待牌名称、役种名称、局况术语要前后一致。
2. 如果项目里已有既定术语，优先沿用项目现有写法。
3. 如果目标语言是英文，要优先采用英语麻将圈中常见、可读的表达；不要逐字硬译成生造英语。
4. 如果某个术语难以自然翻出，可保留罗马字或补最短必要说明，但不要让整页充满混杂写法。

## 元素保留规则

### 非正文文章页

对于以下页面：

- 首页 / index 页
- 站点地图页
- 作者介绍 / 链接集 / 资源推荐页
- 附录导览页

也要使用与正文文章相同的保真标准：

1. 如果原页中的欢迎语、更新履历、区块顺序本身承载信息，就要保留
2. 如果原页是依赖缩略图组织推荐内容，不要把它们压扁成纯文字链接列表
3. 如果内容被拆分到首页和附录，必须保证原始信息完整保留在重建站点中，同时避免重复漂移
4. 这类页面很容易被过度概括，编辑后务必再次回看原始源码

### 图示

如果原文存在独立文章图示，例如：

- `<img src="../images/...">`

要保留它们与周围说明文字的相对位置。

不要把图示挪到脱离上下文的位置。

### 缩略图与推荐图

如果原页包含书封、游戏缩略图或其它推荐图片区块：

1. 要把它们视为承载信息的内容，而不是装饰
2. 保证图片与对应标题、说明的配对关系正确
3. 优先使用本地镜像素材，不要依赖远程热链
4. 如果重建页采用卡片布局，图片区尺寸要统一，保证正文块起点对齐

### 表格

如果原文有表格，要保留：

1. 行列结构
2. 表头含义
3. 数值
4. 表格所表达的警示、排序或强弱关系

如果目标文件是 Markdown，可以使用 Markdown 表格。

### 强调与重要性

要保留如下重要性信号：

1. 粗体关键结论
2. 红色警示语
3. 绿色总结块
4. 高亮的理论句

如果不能逐字保留颜色表现，也要在语义上保留这种强调强度。

## 最终检查

在认为翻译完成之前，必须确认：

1. 没有漏掉原文中的任何例题
2. 没有漏掉任何独立图示
3. 没有漏掉任何表格
4. 没有把粗体、总结、警示内容压平
5. 没有加入代理式旁白
6. 除专有名词或不可避免标签外，没有残留无意义的日文原文
7. 目标语言术语前后一致，没有同一概念多套叫法混用
8. 对于带缩略图的推荐页，本地图片必须正常渲染，卡片与正文对齐也必须稳定

## 输出约束

翻译后的页面必须像页面成品本身，而不是“关于翻译过程的说明文字”。

如果输出为中文：

1. 优先使用自然的中文教学式表达
2. 优先沿用项目内已有中文术语

如果输出为英文：

1. 优先使用自然、可发布的英语教学式表达
2. 不要保留多余的日文句式痕迹
3. 标题、例题名、总结句要像英文站点成品，而不是直译草稿

## 多语言站点落地规则

如果任务不仅是翻译页面，还包含把译文落到多语言站点中，则必须同时考虑以下事项。

### 路由与路径镜像

1. 目标语言页面应尽量与既有主语言页面保持路径镜像关系。
2. 如果项目已经约定语言子目录方案，应严格遵守该方案。
3. 不要擅自改动既有主语言页面的线上路由，只新增目标语言对应路径。
4. 同一批多语言页面中，章节目录、文章层级、前后导航关系应保持一致。

### 页面元数据

1. 目标语言页面应补齐页面级元数据，至少包括：
   - `title`
   - `description`
   - 必要时的 `image`
2. 元数据语言必须与页面正文语言一致。
3. 不要把中文 `description` 直接塞进英文页，也不要把英文标题写进中文页。

### 未完成页面的处理

1. 如果某个目标语言页面只是占位页、骨架页或尚未完成的初稿，应优先使用：
   - `robots: noindex, nofollow`
2. 未完成前，不要把半成品大规模接入主导航、首页推荐区或公开 sitemap。
3. 占位页必须明确标识“建设中”或同等含义，避免被误认为正式成品。

### 语言互链

1. 当同一页面已有两个或以上语言版本时，应建立清晰的互链关系。
2. 如果项目支持 `hreflang`，应为成对页面补齐：
   - 当前语言
   - 对应语言
   - 必要时的 `x-default`
3. 语言切换入口应链接到真正对应的同主题页面，而不是只跳回对方语言首页。

### 导航与用户可见入口

1. 英文版或其他语言版在未形成基本连续阅读能力前，不要提前挂入主导航。
2. 一旦开始公开入口，至少应保证：
   - 首页可访问
   - 栏目页可访问
   - 至少有若干正文页可连续阅读
3. 不要让用户点进某语言版后大量落到空目录或死链接。

### SEO 与索引协同

1. 多语言翻译任务完成后，如果该语言版本准备上线，应同步检查：
   - 页面级 `description`
   - canonical 是否正确
   - `hreflang` 是否成对
   - sitemap 是否包含应公开的目标语言页面
2. 未准备公开的语言版本，不要主动提交到搜索引擎发现机制。
3. 如果项目已使用 `IndexNow` 或其他发现机制，只提交正式可索引的目标语言页面，不要提交占位页和 `noindex` 页面。

## 多语言任务的最终检查

如果本次任务涉及多语言站点落地，在认为完成之前，除常规翻译检查外，还必须确认：

1. 目标语言页面路径与项目多语言路由约定一致
2. 页面元数据语言正确，没有中英混写
3. 未完成页带有正确的 `noindex` 控制
4. 已公开页面没有错误互链、空链接或死链
5. 如果同主题双语页都已存在，语言切换关系和 `hreflang` 准备方案明确
